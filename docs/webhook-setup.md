# Agent Space Webhook 配置

> 本文档描述如何为 DevOps Agent Agent Space 创建 **Generic Webhook**，
> 用于从 GitHub Actions（或其他第三方系统）触发调查。

## 背景

DevOps Agent 有两种被触发方式：

| 方式 | 适用 | 认证 |
|---|---|---|
| `aws devops-agent create-backlog-task` (CLI/SDK) | 仅限已在 Agent Space 内注册的 data plane services（`aws`、`github` association 等） | IAM SigV4 |
| **Agent Space Webhook** (HTTP POST) | **任何外部系统**：GitHub Actions、Grafana、自研脚本 | **HMAC-SHA256** 或 Bearer Token |

本项目走 **Webhook** 路线，因为：

- `create-backlog-task` 的 `reference.system` 字段是服务端白名单（`github` 这种值会被拒绝：`Unknown data plane service name`）
- Webhook 是 AWS 官方推荐给第三方系统的集成点，schema 简单（`eventType`/`priority`/`title`/`description`）
- 不需要 GitHub association，减少一层耦合

## 为什么必须手动在 Console 生成（不能走 Terraform）

截至本文档编写时：

- `aws devops-agent` CLI 只有 `list-webhooks`，**没有 `create-webhook` / `generate-webhook`**
- CloudFormation 没有 `AWS::DevOpsAgent::Webhook` 资源
- 这是安全设计：**HMAC secret 只显示一次**，生成后无法再次读出；UI 创建可以强制用户当场抄录

因此：
- **Webhook URL + Secret**：必须 Console 手动生成（唯一手动步骤）
- **GitHub Actions workflow / secrets / labels**：Terraform GitHub provider 管理
- **AWS 侧所有资源**（Lambda / EventBridge / IAM / SSM）：Terraform AWS provider 管理

## 操作步骤

### 1. 在 Console 打开 Agent Space Webhook 配置向导

进入：
```
https://us-east-1.console.aws.amazon.com/aidevops/home?region=us-east-1
  → Agent Spaces
  → 选中 outline (9e3ca279-9252-4dd8-b19f-1ca9cb7c8439)
  → Capabilities 标签
  → 滚动到 "Webhooks" 板块
  → "Agent Space Webhook" 一行 → 点 "Add webhook"
```

会进入 3 步向导。

### 2. Step 1 — 确认 payload schema

页面会显示 DevOps Agent 期望的 JSON schema：

```typescript
{
  eventType: 'incident';
  incidentId: string;
  action: 'created' | 'updated' | 'closed' | 'resolved';
  priority: "CRITICAL" | "HIGH" | "MEDIUM" | "LOW" | "MINIMAL";
  title: string;
  description?: string;
  timestamp?: string;
  service?: string;
  // The original event generated by service is attached here.
  data?: object;
}
```

⚠️ 后续 GitHub Actions workflow 发送的 payload 必须完全贴合这个 schema，否则 webhook 返回 200 但不会触发 investigation。

点 **Next** 进入 Step 2。

### 3. Step 2 — 确认 HMAC 签名算法

页面展示签名示例（Node.js）：

```javascript
import { createHmac } from "node:crypto";

const payload = { eventType: "incident", /* ... */ };
const timestamp = new Date().toISOString();       // ISO 8601 UTC
const hmac = createHmac("sha256", secret);
hmac.update(`${timestamp}:${JSON.stringify(payload)}`, "utf8");
const signature = hmac.digest("base64");

fetch(webhookUrl, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-amzn-event-timestamp": timestamp,
    "x-amzn-event-signature": signature,
  },
  body: JSON.stringify(payload),
});
```

等效的 Bash + openssl：

```bash
TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%S.000Z)
PAYLOAD='{"eventType":"incident",...}'
SIGNATURE=$(printf '%s:%s' "$TIMESTAMP" "$PAYLOAD" \
            | openssl dgst -sha256 -hmac "$SECRET" -binary | base64)

curl -X POST "$WEBHOOK_URL" \
  -H "Content-Type: application/json" \
  -H "x-amzn-event-timestamp: $TIMESTAMP" \
  -H "x-amzn-event-signature: $SIGNATURE" \
  -d "$PAYLOAD"
```

关键要点：
- 签名内容是 `"${timestamp}:${payload_json}"`（**冒号连接，不是换行**）
- 必须用 `JSON.stringify(payload)` 后的那个**完全相同的 bytes** 作为 body，否则签名对不上
- `timestamp` 必须是 ISO 8601 UTC 格式

点 **Next** 进入 Step 3。

### 4. Step 3 — 生成 URL 和 Secret（⚠️ 只显示一次）

**在点 "Generate URL and secret key" 之前**，先准备好一个终端窗口运行 capture 脚本：

```bash
./scripts/capture-webhook-creds.sh
```

这个脚本会提示你粘贴 URL 和 Secret，然后写入 AWS Secrets Manager 和 GitHub Secrets，避免丢失。

然后在 Console：

1. 点 **Generate URL and secret key** 按钮
2. 页面会显示：
   - **Webhook URL**：形如 `https://event-ai.us-east-1.api.aws/webhook/generic/<UUID>`
   - **Secret key**：一串随机字符串
3. 立刻：
   - 点 **Download .csv file** 保存一份本地备份
   - 或直接复制到 capture 脚本的提示中
4. 勾选 **"I've saved and stored my URL and secret key"**
5. 点 **Add** 完成

创建完成后，页面回到 Capabilities → Webhooks 区域，可以看到新创建的 webhook 列在 "Agent Space Webhook" 下。

## 验证

capture 脚本会自动完成这些，但也可以手动检查：

```bash
# 1. Secrets Manager 里有 webhook 凭据
aws secretsmanager describe-secret \
  --secret-id /devops-agent/agent-space-webhook \
  --region us-east-1

# 2. GitHub secrets 已设置
gh secret list --repo JoeShi/devops-agent-demo-tickets | grep WEBHOOK

# 3. 发一个测试 webhook，应该返回 200
./scripts/verify-pipeline.sh fire
```

## 删除/重建

如果 secret 丢了或需要轮换：

```
Console → 选中 webhook 行 → Actions → Remove
然后重走本文档流程
```

**旧 secret 删除后所有使用它的 workflow 会立刻开始收到 401**，所以轮换窗口期要短。
